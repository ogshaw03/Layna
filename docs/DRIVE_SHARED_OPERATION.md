# Google ドライブ共有アイテム運用のメモ（問題点・対応・API検討）

LayCAT の共有データ（`layna.project.json` / `media/` / `thumbnails/` / `reels/`）を
Google ドライブの共有アイテム／共有ドライブに置いて複数人で運用したときの、
既知の問題・現状の対応・Drive API/Firebase 検討をまとめる。

## 前提（現状アーキテクチャ）
- File System Access API でフォルダを直接読み書き。
- 共有は「同じフォルダを各自が接続」。**ライブ同期・外部更新の監視は無い**（`boot`/接続時にのみ読込）。
- `persist()` は在メモリ DB を丸ごと `layna.project.json` に書き出す。

## 既知の問題（重大度順）

### 🔴 高
1. **`layna.project.json` の競合＝他者の変更が消える**
   Drive は JSON をマージしない。同時編集で「競合コピー」が作られ、LayCAT は正規名しか読まないため
   片方の変更が失われる。`persist()` の丸ごと上書き＋ライブ更新なしが要因。
2. **「閲覧者／コメント可」で共有されていると書き込めない**
   共有アイテムが編集権限でないとローカルは実質読み取り専用 → `createWritable` が
   `NoModificationAllowedError`。**リトライでは回復しない（権限不足）。共有は「編集者」必須。**

### 🟡 中
3. **メディア/サムネが解決できずプレースホルダになる（無言失敗）**
   別ユーザーのアップロードで JSON が先に届き実体（mp4/jpg）が未同期、または
   Drive「ストリーミング（オンラインオンリー）」で `getFile()` が読めない。失敗が無言でプレースホルダ表示。
4. **フォルダハンドル/マウントパスの不安定さ**
   共有アイテムのドライブ文字・マウント位置が変わるとハンドルが無効化 → 再接続が必要。
   （起動時は queryPermission のみ＝黙って未接続＝データが消えたように見える → 保存先表示から再接続）

### 🟢 低
5. **大容量動画のストリーミング負荷**（ホバー再生・比較再生がオンデマンドDLを誘発）。
6. **`reels/reels.json` も #1 と同じ競合・遅延問題**。

## 現状の対応（実装済み）
- **#1 競合対策（`v2026.07.09.38`）**：`persist` 保存前にフォルダ側の現在値を再読込し、
  自分が最後に同期した内容（baseline）と異なれば **3-way マージ**（`_merge3`：ノード/サブミット単位、
  自分の編集優先・相手の追加/編集も保持・削除は相手が触っていなければ反映）。
  マージ時は**元のフォルダ内容を `layna.project.conflict-<時刻>.json` にバックアップ**し、
  結果を DB に反映してトースト通知。暗号化プロジェクトは復号して比較・再暗号化して保存。
  ※ 同一ノードを同時編集した場合は自分優先（相手のその編集は失われるが、バックアップに残る）。ライブ同期ではない。
- **#3 メディア/サムネのリトライ（`v2026.07.09.38`）**：`resolveThumbs` は未解決分をバックオフで自動リトライ
  （1.2s→3s→6s→12s→20s）。その間は「同期待ち」表示（`.thumb-wait`）。レビュー窓の動画も `getURLRetry` で
  数回リトライし、だめならトースト。
- **書き込み失敗の耐性（`v2026.07.09.36-37`）**：`persist` の直列化（同一ファイルの並行 `createWritable` 競合防止）、
  `storage._write` で失敗時 `abort`（ロック解放）＋一時ロックのリトライ、`putMedia` の try/catch。

## 未対応・限界
- **ライブ同期は無い**（他者の変更は次の保存時マージ or リロードで反映）。
- **同一ノードの同時編集**は自分優先（相手の編集はバックアップ止まり）。
- **権限不足（閲覧者）** はアプリ側では書けない（運用で編集者権限にするしかない）。
- **オンラインオンリー**の完全な解消は不可（リトライで緩和のみ）。→ Drive は「ミラーリング」推奨。

## Drive API / Firebase の検討
| | 現状：FSA＋Driveデスクトップ | Google Drive API | Firebase（Firestore＋Storage） |
|---|---|---|---|
| #1 競合/消失 | △ 保存時3-wayマージで緩和 | △ 楽観ロックは作れるがマージは自前 | ✅ フィールド単位＋リアルタイムで本質解決 |
| #2 権限 | ❌ OSのRO化 | ✅ OAuthで明確 | ✅ セキュリティルール |
| #3 メディア同期 | △ リトライで緩和 | △ DL URL（大容量は帯域） | ✅ 安定URL・再開可能アップロード |
| リアルタイム | ❌ | ❌（ポーリング/Webhook自前） | ✅ 標準 |
| 動画ストリーミング | △ Drive次第 | ❌ OAuth配信が面倒（要プロキシ） | ✅ 署名URL/レンジ |
| 実装コスト | 済 | 中〜大 | 中 |
| データ置き場 | 顧客Drive | 顧客Drive | Firebase |

### 結論・方針
- **Drive API 単独は非推奨**：協働の要（realtime/merge）が手に入らず、労力の割に効果が薄い。
- **同時編集する運用なら Firebase（Firestore＋Storage）移行が本命**（認証は既に `laycat-54ee4` で稼働）。
  realtime が #1 を根本から消し、Storage が #3 を解消、ルールが #2 を解決。
- **素材を必ず Drive に置く要件がある場合**：Drive API、または「メタデータ=Firestore／動画=Drive」のハイブリッド。
- **当面**：現状＋保存時マージ・リトライ（実装済み）で延命し、手狭になったら Firebase へ段階移行。

### Drive 運用のおすすめ設定
- 共有は必ず**「編集者」**権限。
- Drive for Desktop は**「ミラーリング（ローカルにコピー）」**（ストリーミングだとオンラインオンリーで読めないことがある）。
- できれば**「共有ドライブ」**を使う（「共有アイテム＝Shared with me」は同期・オフライン挙動が制限的）。
- 同時刻の同一プロジェクト編集はできるだけ避ける（自動マージはするが同一ノード同時編集は自分優先）。
