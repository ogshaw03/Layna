# A/B 比較再生 — 現状と今後の改善メモ

## 現状（実装済み）
アノテーションウィンドウ（`openReview`）内に B ペインを横並び表示し、A と同期再生する。

### 同期方式（重要な前提）
- **アニメーション確認のため、再生速度は絶対に変えない**という制約がある。
- そのため **A を唯一のマスター**とし、A は native 再生（速度不変）。
- **B は自前で再生させず**、A の現在時刻へ **毎フレーム精密シーク**（`bVid.currentTime = media.currentTime`）して同一フレームを表示する。
- `seeked` 完了を待ってから次のシークを積む（`bSeekBusy` でキュー詰まりを防止）。
- 停止・シーク・コマ送り時は従来どおりフレーム厳密一致。
- 実装箇所: `openReview` 内の `cmpFrameSync` / `syncBToA` / `enterCompare` / `exitCompare`、rAF ループ `sync()` からの呼び出し。

### 既知の限界
- B は「毎フレーム、A の位置へジャンプ」する形で更新されるため、**B 側の表示がごく僅かにカクつく**ことがある（A の再生には影響しない）。
- 原因はブラウザの seek スループット（1フレームごとの精密 seek のデコード待ち）。ユーザー了承済みで一旦 OK としている（2026-07）。

---

## 今後の改善案：フレームキャッシュ方式（B を滑らかに）
再生速度を変えないまま B の表示を滑らかにするための案。

### 方針
- B の各フレームを**事前にデコードして画像としてキャッシュ**し、A の時計（マスタークロック）に合わせて `<canvas>` に描画する。
- これにより B は seek 待ちなしで即時にフレームを差し替えられ、A と完全同一フレーム・速度不変・滑らかを同時に満たせる。

### 実装アイデア
1. **`requestVideoFrameCallback` で抽出**: B を一度 muted で通し再生（またはオフスクリーンで seek 走査）し、各フレームを `createImageBitmap` / `OffscreenCanvas` に取り込みフレーム番号→ImageBitmap の Map を作る。
2. **表示は canvas**: B ペインを `<video>` ではなく `<canvas>` にし、A の rVFC / rAF で得た現在フレーム番号のキャッシュ画像を描画。
3. **メモリ対策**: レビュークリップは短尺想定だが、長尺・高解像度に備えて
   - 解像度を表示サイズに合わせて縮小（比較表示は等倍不要なことが多い）、
   - 上限フレーム数を超える場合は間引き or LRU で破棄、
   - 事前抽出が重い場合は再生範囲だけ遅延キャッシュ（プリフェッチ）。
4. **フォールバック**: `requestVideoFrameCallback` 非対応環境や抽出失敗時は現行の精密 seek 方式に自動フォールバック。

### 検討事項 / リスク
- 事前抽出中のローディング表示（「B を読み込み中…」）が必要。
- 大量フレームのキャッシュはメモリを食う → 上限とクリップ長のガードを設ける。
- A 側もキャッシュ化すれば完全に同一クロックで駆動できるが、A はアノテーション描画のため video 要素を維持したい → A は video のまま、B のみ canvas キャッシュが現実的。
- fps が A と B で異なるケースの対応（現状は「同一タイムコード」で合わせている。フレーム番号ベースにするなら fps 変換が必要）。

### 代替（軽量）案
- キャッシュせずとも、B の seek に `requestVideoFrameCallback` を併用して presented frame のタイミングで描画すると、体感のカクつきが軽減する可能性がある（要検証）。
- B の解像度を落とした比較用プロキシ動画を用意しておくと seek が速くなり、現行方式のままでも滑らかになりうる。
